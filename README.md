# 抖音作品数据转换工具说明

## 1. 交付文件

`index` 文件夹是可交付网页包：

- `index/index.html`：直接双击或用浏览器打开的主页面，已经内嵌 CSS 和 JS，不依赖本地服务。
- `index/assets/app.js`：构建出的脚本备份，便于排查。
- `index/assets/app.css`：构建出的样式备份，便于排查。
- `index/README.md`：本说明文档。

日常使用优先打开：

```text
index/index.html
```

## 2. 工具用途

将抖音后台导出的单条作品原始 Excel 数据，转换为最终统计表中 `K-AJ` 区间的数据。工具全部在本地浏览器运行，不上传文件到服务器。

支持上传：

- 单个作品的一组 Excel 文件。
- 多个作品文件夹，一次批量处理。

工具会自动识别文件内容，不依赖文件名。

## 3. 输入文件要求

每个作品至少需要以下累计每小时趋势表：

- `播放量-累计-每小时趋势数据`
- `点赞量-累计-每小时趋势数据`
- `评论量-累计-每小时趋势数据`
- `分享量-累计-每小时趋势数据`
- `收藏量-累计-每小时趋势数据`
- `涨粉量-累计-每小时趋势数据`

视频作品还需要：

- `完播率-累计-每小时趋势数据`

如果上传的是 `新增-每小时`、`新增-每天`、`累计-每天` 或其他非累计每小时数据，工具会提示原始数据有误，并阻止导出。

## 4. 发布时间识别

工具默认使用各趋势表第一条数据的小时作为发布时间小时。

例如趋势表第一行时间是：

```text
2026-05-01 10:00
```

则发布时间小时为：

```text
2026-05-01 10:00
```

页面上可以手动修改“发布时间小时”。修改后，所有 `24h`、`48h`、`7天` 字段会按新的时间重新计算。

## 5. 计算逻辑

### 5.1 时间窗口

本工具采用方案 A：

- `24h`：发布时间小时 + 24 小时。
- `48h`：发布时间小时 + 48 小时。
- `7天`：发布时间小时 + 168 小时。

工具要求趋势表中必须存在对应小时的数据。如果缺少目标小时，会报错。

### 5.2 播放、互动、收藏类指标

因为输入要求是累计每小时数据，所以直接取目标小时的累计值。

计算公式：

```text
24h:播放量 = 播放量累计趋势表[发布时间小时 + 24h]
48h:播放量 = 播放量累计趋势表[发布时间小时 + 48h]
7天:播放量 = 播放量累计趋势表[发布时间小时 + 168h]
```

点赞、评论、分享、收藏同理：

```text
24h:点赞量 = 点赞量累计趋势表[发布时间小时 + 24h]
48h:点赞量 = 点赞量累计趋势表[发布时间小时 + 48h]
7天:点赞量 = 点赞量累计趋势表[发布时间小时 + 168h]
```

评论、分享、收藏按相同规则计算。

### 5.3 增粉

增粉来自 `涨粉量-累计-每小时趋势数据`。

最终表中字段名叫“增粉”，原始文件中字段名通常叫“涨粉量”。

计算公式：

```text
24h:增粉 = 涨粉量累计趋势表[发布时间小时 + 24h]
48h:增粉 = 涨粉量累计趋势表[发布时间小时 + 48h]
7天:增粉 = 涨粉量累计趋势表[发布时间小时 + 168h]
```

### 5.4 完播率

完播率只对视频作品输出。

如果检测到 `完播率-累计-每小时趋势数据`，则按目标小时取累计完播率：

```text
24h:完播率 = 完播率累计趋势表[发布时间小时 + 24h]
48h:完播率 = 完播率累计趋势表[发布时间小时 + 48h]
7天:完播率 = 完播率累计趋势表[发布时间小时 + 168h]
```

如果没有完播率趋势表，工具认为该作品没有可用完播率来源，完播率列留空。

### 5.5 公式列

以下字段属于最终表中的公式列，工具页面会预览，但导出时不作为手填值输出：

```text
24h:互动总量 = 24h:点赞量 + 24h:评论量 + 24h:分享量 + 24h:收藏量
48h:互动总量 = 48h:点赞量 + 48h:评论量 + 48h:分享量 + 48h:收藏量
7天:互动总量 = 7天:点赞量 + 7天:评论量 + 7天:分享量 + 7天:收藏量
7天互动率 = 7天:互动总量 / 7天:播放量
```

对应列：

- `P`：24h:互动总量
- `X`：48h:互动总量
- `AF`：7天:互动总量
- `AJ`：7天互动率

### 5.6 暂无来源字段

`AI：7天:作品预约组件留资量` 当前原始文件中没有明确来源，工具默认留空。

## 7. 技术实现

### 7.1 技术栈

- React：页面交互。
- TypeScript：类型约束。
- xlsx：在浏览器本地解析和导出 Excel。
- esbuild：打包成可直接打开的单文件网页。

### 7.2 核心源码

项目源码在上一级目录：

```text
src/converter.ts
src/App.tsx
src/main.tsx
src/styles.css
scripts/build-standalone.mjs
```

核心规则集中在：

```text
src/converter.ts
```

主要结构：

- `METRIC_LABEL`：指标名映射。
- `OUTPUT_COLUMNS`：最终表 `K-AJ` 的列定义。
- `parseUploadGroup()`：解析上传文件，识别趋势表，校验文件类型。
- `buildGroupResult()`：按发布时间和时间窗口计算输出值。
- `exportResults()`：生成导出 Excel。

### 7.3 文件识别方式

工具通过 sheet 名和第二列表头识别指标。

例如：

```text
播放量-累计-每小时趋势数据
点赞量-累计-每小时趋势数据
涨粉量-累计-每小时趋势数据
完播率-累计-每小时趋势数据
```

识别逻辑在：

```text
detectMetric()
detectMode()
detectGranularity()
```

## 8. 构建与维护

### 8.1 本地开发

安装依赖后运行：

```bash
npm run dev
```

浏览器访问：

```text
http://127.0.0.1:5173/
```

### 8.2 重新打包网页

修改源码后运行：

```bash
npm run build
```

构建会生成：

```text
dist/index.html
dist/assets/app.js
dist/assets/app.css
```

当前构建脚本会把 CSS 和 JS 内嵌到 `dist/index.html`，所以 `dist/index.html` 可以直接用浏览器打开。

如果要更新交付包，重新构建后复制：

```text
dist/index.html -> index/index.html
dist/assets/app.js -> index/assets/app.js
dist/assets/app.css -> index/assets/app.css
```

### 8.3 修改计算规则

常见维护位置：

- 新增或删除输出列：修改 `src/converter.ts` 中的 `OUTPUT_COLUMNS`。
- 改 7天规则：修改 `buildGroupResult()` 中对 `168` 小时窗口的处理。
- 增加新指标识别：修改 `MetricKey`、`METRIC_LABEL`、`detectMetric()`。
- 修改公式列：修改 `interactionKeys()` 和 `formulaPreview` 相关逻辑。

### 8.4 常见问题

如果页面打开空白：

1. 优先打开 `index/index.html`。
2. 如果打开的是项目根目录的 `index.html`，它会自动跳转到 `index/index.html`。
3. 如果修改过源码但没有重新构建，运行 `npm run build`。

如果导出按钮不可用：

1. 查看页面上每组作品的错误提示。
2. 确认上传的是 `累计-每小时趋势数据`。
3. 确认所有必需指标都存在。
4. 确认趋势表覆盖了 `发布时间小时 + 168h`。

如果 7天数据和人工表不一致：

当前规则是严格按 `发布时间小时 + 168h` 取值，不取导出时最新累计值。
